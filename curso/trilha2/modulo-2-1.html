🔌 O que é MCP

O Model Context Protocol (MCP) é um protocolo aberto criado pela Anthropic que define como assistentes de IA se conectam a dados e ferramentas externas. Em vez de cada agente ter sua própria integração proprietária com cada serviço, o MCP estabelece um padrão universal — qualquer agente pode se conectar a qualquer ferramenta que implemente o protocolo.

Conceito Principal

A melhor analogia para entender o MCP:

HTTP

Conecta browsers a servidores web. Qualquer browser acessa qualquer site.

=
MCP

Conecta agentes a ferramentas. Qualquer agente acessa qualquer tool.

Antes do MCP, cada integração era um caso especial. Se o Claude queria acessar o GitHub, precisava de um plugin específico. Se o Gemini queria acessar o Slack, outra integração. Com o MCP, basta que o GitHub tenha um MCP server e qualquer agente com um MCP client pode interagir com ele. O protocolo elimina o problema NxM (N agentes x M ferramentas) e transforma em N+M.

💡 Por que isso importa

O MCP foi lançado em novembro de 2024 e em menos de dois anos já é suportado nativamente por Claude Code, Gemini CLI, Cursor, VS Code, Zed, e dezenas de outras ferramentas. A adoção acelerada mostra que a indústria precisava desesperadamente de um padrão aberto para conectividade de agentes.

🏗️ Arquitetura MCP

A arquitetura do MCP segue um modelo client-server com três componentes fundamentais: o Client (agente), o Server (ferramenta) e o Transport (camada de comunicação). Entender esses componentes é essencial para trabalhar com qualquer integração MCP.

Diagrama da Arquitetura

MCP Client

Claude Code, Gemini CLI,
Cursor, VS Code...

discover → negotiate → ← results
Transport

stdio (local)
HTTP/SSE (remoto)

→ call tool → read resource ← response
MCP Server

GitHub, Slack, DB,
filesystem...

O fluxo começa com o client enviando um initialize request para o server, que responde com suas capabilities (quais tools, resources e prompts oferece). Depois da negociação inicial, o client pode chamar tools, ler resources e usar prompts conforme necessário. Toda a comunicação usa JSON-RPC 2.0.

📊 Transports disponíveis

stdio — Comunicação via stdin/stdout. Ideal para servers locais que rodam como processos. O client spawna o server como subprocess.
HTTP + SSE — Server-Sent Events para streaming. Ideal para servers remotos hospedados em cloud. Suporta autenticação OAuth.
Streamable HTTP — Evolução do SSE para 2026, com suporte bidirecional. Permite servers stateless e escaláveis.

📋 Resources e Tools

O MCP define dois tipos fundamentais de capacidades que um server pode expor: Resources (dados para leitura) e Tools (ações que podem ser executadas). Essa separação é intencional e reflete a diferença entre consultar informação e agir no mundo.

📖 Resources

  • •Read-only: apenas expõem dados, nunca modificam estado
  • •URI-based: cada resource tem um URI único (ex: git://repo/main)
  • •Tipados: retornam text/plain, application/json, ou binário
  • •Listáveis: o client pode descobrir todos os resources disponíveis
  • •Subscrevíveis: o client pode ser notificado quando mudam

🔧 Tools

  • •Executáveis: realizam ações que podem modificar estado
  • •Parâmetros tipados: JSON Schema define inputs esperados
  • •Model-controlled: o LLM decide quando e como chamar
  • •Descritivas: incluem descrição para o LLM entender o propósito
  • •Confirmáveis: o client pode pedir aprovação humana antes de executar

Exemplo: MCP Server de Git

Um MCP server para Git exporia a seguinte estrutura:

Resources (dados)

  • git://repos → lista de repositórios
  • git://repo/main/files → árvore de arquivos
  • git://repo/main/log → histórico de commits
  • git://repo/status → status do working tree

Tools (ações)

  • git_commit → criar um commit
  • git_push → enviar para remote
  • git_branch → criar/trocar branch
  • git_diff → gerar diff entre refs

🔄 Prompts e Sampling

Além de resources e tools, o MCP define duas capacidades avançadas que tornam as interações mais sofisticadas: Prompts (templates reutilizáveis) e Sampling (geração bidirecional de texto). Esses mecanismos permitem workflows guiados e chain-of-thought estruturado.

Prompts: Templates reutilizáveis

Um MCP server pode oferecer prompts pré-construídos que guiam o agente em tarefas complexas. O client lista os prompts disponíveis e pode invocar qualquer um com argumentos.

Exemplo: prompts de um server de code review

  • review_pr(pr_number) → template completo para revisar um PR
  • explain_code(file_path) → template para explicar código
  • suggest_tests(function_name) → template para sugerir testes

Sampling: O server pede ao client que gere texto

Sampling inverte o fluxo normal: em vez de o client chamar o server, o server pede ao client que use o LLM para gerar texto. Isso permite que o server orquestre raciocínios complexos sem ter acesso direto ao modelo.

Caso de uso: um MCP server de deploy detecta um erro. Em vez de apenas retornar o erro, ele pede ao client que analise o log de erro e sugira uma correção. O server então pode aplicar a sugestão automaticamente. Isso cria um loop de raciocínio poderoso onde o server e o LLM colaboram.

💡 Dica Prática

Sampling é a funcionalidade menos adotada do MCP em 2026, mas potencialmente a mais transformadora. Ela permite que ferramentas "pensem" usando o LLM do client, criando uma camada de inteligência distribuída. Fique de olho em servers que usam sampling para workflows multi-step.

🗺️ Roadmap MCP 2026

O MCP está amadurecendo rapidamente. A especificação evoluiu de um draft inicial em novembro de 2024 para um protocolo robusto adotado por toda a indústria. O roadmap de 2026 foca em escalabilidade, interoperabilidade entre agentes e governança enterprise.

Evolução do MCP

Nov 2024 — Lançamento

Anthropic publica a spec inicial. Suporte em Claude Desktop. Community adopta rapidamente.

Q1 2025 — Adoção ampla

Cursor, VS Code, Zed adotam MCP. Centenas de servers open-source surgem. Streamable HTTP transport.

Q3 2025 — Enterprise features

OAuth 2.1, authorization framework, audit logging. Grandes empresas começam a adotar.

2026 — Maturidade

Streaming, paginação, agent-to-agent communication, registry/discovery, governança enterprise.

📊 Prioridades do Roadmap 2026

Streaming & Paginação

Suportar datasets grandes e respostas incrementais sem timeout

Agent-to-Agent

Agentes comunicando entre si via MCP para tarefas distribuídas

Registry & Discovery

Catálogo centralizado de servers com busca, avaliação e trust scores

Governança Enterprise

RBAC, compliance, audit trails, aprovações multi-nível

🐳 Docker MCP Gateway

A Docker entrou no ecossistema MCP de forma estratégica, criando um catálogo, toolkit e gateway para MCP servers. A proposta é tratar MCP servers como containers — com todo o isolamento, portabilidade e versionamento que isso implica.

Docker MCP Toolkit

O Docker MCP Toolkit é composto por três peças:

1.
Catalog — Repositório curado de MCP servers empacotados como imagens Docker. Similar ao Docker Hub, mas específico para MCP. Inclui metadata, versões e trust scores.
2.
Toolkit — CLI e SDK para criar, testar e publicar MCP servers como containers. Inclui templates, hot-reload para desenvolvimento e CI/CD integration.
3.
Gateway — Proxy que gerencia múltiplos MCP servers, faz load balancing, autenticação centralizada e logging. Um único ponto de entrada para todos os servers.

⚠️ MCP como Infraestrutura

O movimento do Docker sinaliza uma mudança fundamental: MCP servers estão deixando de ser "plugins" e se tornando infraestrutura. Assim como containers transformaram deploy de aplicações, MCP containers podem transformar deploy de capacidades de IA. Empresas que ignorarem isso podem ficar para trás.

As vantagens de rodar MCP servers como containers são significativas: isolamento (cada server roda em seu próprio sandbox), portabilidade (funciona igual em qualquer máquina), versionamento (tags e rollback), e escalabilidade (orquestração via Kubernetes). Isso resolve muitos dos problemas de segurança e deployment que MCP servers nativos enfrentam.

Resumo Final

✓ MCP é o protocolo aberto que padroniza a comunicação entre agentes de IA e ferramentas externas
✓ A arquitetura client-server com transports flexíveis (stdio, HTTP/SSE) suporta cenários locais e remotos
✓ Resources fornecem dados read-only; Tools executam ações com parâmetros tipados
✓ Prompts e Sampling adicionam templates reutilizáveis e geração bidirecional de texto
✓ O roadmap 2026 foca em escalabilidade, agent-to-agent e governança enterprise
✓ Docker MCP Gateway trata MCP servers como infraestrutura containerizada
