Extensões, plugins e skills: a nova camada

Descubra como os agentes CLI modernos se tornam extensíveis através de plugins, extensões, skills e hooks — uma nova camada de composição que vai muito além dos plugins tradicionais.

🧩 O modelo de extensibilidade

Plugins clássicos seguem um modelo simples: o host define uma interface, o plugin a implementa, e o host carrega o plugin em runtime. Mas agentes de IA precisam de algo mais sofisticado. A extensibilidade moderna combina ferramentas (MCP), prompts, comandos, hooks e sub-agentes em uma camada nova de composição.

Por que plugins clássicos não bastam

O modelo clássico de plugin tem três limitações fundamentais quando aplicado a agentes:

Estático

Plugins tradicionais expõem uma API fixa. Agentes precisam descobrir capabilities dinamicamente e adaptar seu comportamento.

Unidirecional

O host chama o plugin. Mas agentes precisam de comunicação bidirecional — o plugin pode precisar que o agente "pense".

Sem contexto

Plugins não sabem o que o agente está fazendo. Extensões modernas fornecem e consomem contexto continuamente.

📊 Camadas de extensibilidade em 2026

Tools (MCP) — Ferramentas que o agente pode chamar: ler arquivo, executar query, criar commit
Prompts & Skills — Conhecimento específico de domínio que guia o agente em tarefas especializadas
Comandos — Slash commands e atalhos que o usuário invoca diretamente
Hooks — Callbacks que executam em eventos específicos (pre-commit, post-edit, on-error)
Sub-agents — Agentes especializados que o agente principal pode delegar tarefas

📦 Extensões do Gemini CLI

O Gemini CLI introduziu um sistema de extensões que empacota múltiplos componentes em uma unidade coesa. Uma extensão Gemini pode conter prompts, MCP servers, comandos customizados, hooks e skills — tudo definido em um único arquivo JSON de configuração.

Anatomia de uma extensão Gemini

Uma extensão é um pacote que declara:

1. MCP servers que devem ser iniciados — com configuração de transport e args
2. System prompts adicionais que contextualizam o agente para o domínio
3. Hooks que executam em pontos específicos do ciclo de vida
4. Metadata — nome, versão, autor, descrição para discovery

O ecossistema de extensões do Gemini CLI é aberto — qualquer desenvolvedor pode criar e distribuir extensões. A Google trabalha com parceiros para criar extensões oficiais para serviços populares (Firebase, Cloud Run, BigQuery), mas a comunidade já produziu centenas de extensões para uso geral.

💡 Dica Prática

Para instalar uma extensão no Gemini CLI, basta adicionar ao arquivo de configuração extensions.json na raiz do projeto. O Gemini CLI detecta automaticamente e inicializa os MCP servers e hooks declarados. Sem restart necessário.

3

🔧 Plugins do Claude Code

O Claude Code usa MCP servers como backbone de extensibilidade. Em vez de criar um sistema de plugins próprio, a Anthropic apostou no próprio protocolo que criou — e isso significa que qualquer MCP server funciona como "plugin" do Claude Code.

Mecanismos de extensão do Claude Code

MCP Servers

Configurados em .claude/settings.json. Adicionam tools e resources que o agente pode usar. Suportam stdio e HTTP.

Slash Commands

Comandos como /review, /commit, /test. Podem ser built-in ou criados pelo usuário via skills.

Hooks

Scripts que executam em eventos: pre-tool-use, post-tool-use, notification. Permitem automação e validação.

CLAUDE.md

Arquivo de contexto por projeto que define regras, convenções e instruções para o agente seguir.

📊 Configuração típica de MCP no Claude Code

O arquivo .claude/settings.json na raiz do projeto define quais MCP servers ativar:

{ "mcpServers": {

"github": { "command": "mcp-server-github" },

"postgres": { "command": "mcp-server-postgres",

"args": ["postgresql://localhost/mydb"] },

"filesystem": { "command": "mcp-server-filesystem",

"args": ["/home/user/docs"] }

} }

📝 SKILL.md — O contrato entre agente e ferramenta

O SKILL.md é um arquivo Markdown que descreve o que uma ferramenta faz, como usá-la e quais comandos estão disponíveis. É o contrato legível por máquinas e humanos entre um agente e uma ferramenta — o LLM lê o SKILL.md para saber como operar qualquer ferramenta.

O que um SKILL.md contém

1
Descrição

O que a ferramenta faz, em linguagem natural. O LLM usa isso para decidir quando invocar a skill.

2
Comandos disponíveis

Lista de comandos, flags e parâmetros. Formatado para o LLM entender a sintaxe.

3
Exemplos de uso

Cenários concretos com input e output esperado. Funciona como few-shot examples para o LLM.

4
Restrições e advertências

O que não fazer, limites conhecidos, cuidados de segurança.

💡 CLI-Anything e SKILL.md automático

O conceito de CLI-Anything (Trilha 3) leva o SKILL.md ao limite: qualquer software com interface CLI pode ter seu SKILL.md gerado automaticamente. O agente analisa o --help, man pages e exemplos de uso para criar um SKILL.md completo. Isso transforma qualquer CLI em uma ferramenta utilizável por agentes sem precisar de adaptação manual.

🏪 Catálogos e marketplaces

Com milhares de MCP servers e extensões disponíveis, a descoberta e curadoria se tornam problemas críticos. Vários catálogos e marketplaces emergiram para organizar, avaliar e distribuir extensões para agentes CLI.

📊 Principais catálogos em 2026

Docker MCP Catalog

Repositório oficial de MCP servers como containers Docker. Curado, versionado, com trust scores. O mais focado em segurança e enterprise.

Smithery

Marketplace comunitário para MCP servers. Interface web para descoberta, com reviews e documentação. Foco em developer experience.

mcp.run

Plataforma que roda MCP servers como WebAssembly em sandbox seguro. Elimina necessidade de instalar dependências locais.

⚠️ Cuidados com catálogos

Assim como npm e Docker Hub, catálogos de MCP servers enfrentam desafios de segurança. Um server malicioso pode exfiltrar dados, executar código perigoso ou servir como vetor de supply chain attack. Sempre verifique: quem é o autor? O código é open-source? Tem reviews? O catálogo faz scan de segurança?

🔗 Interoperabilidade

Uma das maiores promessas do MCP é a interoperabilidade: extensões criadas para um ecossistema funcionando em outro. Na prática, o MCP funciona como lingua franca, mas existem limites importantes que você precisa conhecer.

✅ O que funciona cross-platform

  • •MCP Tools: qualquer MCP server funciona em qualquer client que suporte MCP
  • •Resources: dados expostos via MCP são universalmente acessíveis
  • •Transport: stdio e HTTP funcionam em qualquer implementação

❌ Limitações

  • •Hooks: sistema de hooks do Gemini ≠ hooks do Claude Code
  • •Auth: cada plataforma pode usar mecanismo diferente de autenticação
  • •Features proprietárias: sampling, prompts avançados podem ter suporte variável

O caminho para interoperabilidade total

O MCP como protocolo garante que a camada de tools e resources é interoperável. Mas extensões completas — que incluem hooks, prompts customizados e configuração de UI — ainda são específicas de cada plataforma. A tendência é que a spec MCP absorva mais dessas funcionalidades ao longo de 2026, aumentando a portabilidade. Enquanto isso, a melhor estratégia é focar extensões na camada MCP e minimizar dependências proprietárias.

Resumo Final

✓ A extensibilidade de agentes combina tools, prompts, comandos, hooks e sub-agentes
✓ Gemini CLI usa extensões empacotadas em JSON; Claude Code usa MCP servers como backbone
✓ SKILL.md é o contrato entre agente e ferramenta — legível por humanos e máquinas
✓ Catálogos como Docker MCP, Smithery e mcp.run organizam a descoberta de extensões
✓ MCP garante interoperabilidade na camada de tools; features proprietárias ainda limitam portabilidade total
