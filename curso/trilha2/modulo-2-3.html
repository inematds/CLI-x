Sandboxing, segurança e governança

Agentes com acesso a shell, arquivos e rede são poderosos — e perigosos. Entenda os modelos de sandboxing, políticas de execução, auditabilidade e como a indústria está lidando com os riscos reais de segurança.

1

🔒 Por que sandboxing importa

Quando você dá a um agente de IA acesso ao shell do seu computador, está concedendo poder equivalente ao de um administrador de sistema. Sem sandbox, um prompt injection ou um MCP server malicioso pode executar rm -rf /, exfiltrar dados sensíveis ou instalar malware. Segurança não é opcional — é pré-requisito.

⚠️ Cenários de risco real

1. Prompt injection via dados: o agente lê um arquivo que contém instruções maliciosas embutidas. O LLM segue as instruções e executa comandos destrutivos.
2. MCP server comprometido: um server de terceiro retorna tool results com instruções injetadas que o agente executa cegamente.
3. Exfiltração silenciosa: o agente envia conteúdo de arquivos sensíveis (.env, SSH keys) para um endpoint externo via curl.
4. Escalação de privilégio: o agente modifica configurações do sistema para ganhar acesso permanente.

Princípio fundamental

O princípio de segurança para agentes CLI é simples: assume breach. Projete seu sistema assumindo que o agente será comprometido em algum momento. Sandboxing garante que, mesmo quando isso acontecer, o dano seja contido e reversível.

2

📦 Modelos de sandbox

Existem múltiplas abordagens para isolar agentes CLI, cada uma com trade-offs diferentes entre segurança, performance e usabilidade. Os agentes modernos combinam várias dessas técnicas em camadas.

📊 Comparação de modelos

Container (Docker) Alta segurança

Isolamento completo de filesystem, rede e processos. Usado pelo Codex CLI com sandbox obrigatório. Overhead: ~100ms startup, ~50MB memória.

Namespace (Linux) Média-alta segurança

Usa namespaces do kernel para isolar PID, network, mount. Mais leve que container. Não disponível em macOS sem VM.

Filesystem (chroot/overlay) Média segurança

Restringe acesso ao filesystem a diretórios específicos. Overlay FS permite "undo" de mudanças. Usado por Gemini CLI sandbox mode.

Aprovação humana Depende do humano

O agente mostra o que quer fazer e espera OK do usuário. Usado pelo Claude Code para edições e comandos shell. Nenhum isolamento técnico.

3

🛡️ Políticas de execução

Políticas de execução definem o que o agente pode e não pode fazer, usando allowlists e blocklists de comandos, categorias de risco e regras configuráveis por projeto ou por agente.

Categorias de comandos

Leitura (Safe)

ls, cat, grep, find, git status, git log. Auto-aprovado. Não muda estado.

Escrita (Needs Approval)

mkdir, touch, git commit, npm install. Muda estado local. Requer confirmação.

Rede (Restricted)

curl, wget, ssh, git push. Pode exfiltrar dados ou acessar sistemas remotos. Requer confirmação + logging.

Destrutivo (Blocked)

rm -rf, chmod 777, sudo, dd, mkfs. Potencialmente irreversível. Bloqueado por padrão.

💡 Configuração por projeto

A melhor prática é definir políticas por projeto. Um projeto de frontend pode permitir npm run build automaticamente, enquanto um projeto de infraestrutura pode bloquear qualquer comando que altere configuração de rede. Claude Code permite isso via .claude/settings.json e Gemini CLI via arquivo de configuração do projeto.

4

🔍 Auditabilidade

Auditabilidade é a capacidade de saber exatamente o que o agente fez: quais comandos executou, quais arquivos leu ou modificou, quais tools MCP chamou e quais resultados obteve. Em ambientes regulados, uma trilha auditável completa é obrigatória.

O que um audit trail completo inclui

1.
Sessão

Timestamp, usuário, projeto, modelo utilizado, configurações ativas

2.
Comandos executados

Cada comando shell com input, output, exit code e duração

3.
Arquivos modificados

Diff completo de cada arquivo alterado (before/after)

4.
Tools MCP chamadas

Qual server, qual tool, parâmetros enviados, resultado recebido

5.
Decisões humanas

Quando o humano aprovou/rejeitou, o que foi mostrado ao humano

📊 Reprodutibilidade

Um bom audit trail permite reprodutibilidade: dado o log completo, deveria ser possível refazer exatamente o que o agente fez. Isso é essencial para debugging, compliance e para treinar equipes. Claude Code grava logs de sessão em ~/.claude/logs/; o Codex CLI mantém histórico completo de cada sandbox.

⚠️ Falhas reais

Segurança de agentes não é teórica — já houve falhas reais e documentadas que mostram os riscos concretos de conectar LLMs a ferramentas externas sem proteção adequada.

Caso: MCP Server de Git da Anthropic

Em 2025, pesquisadores de segurança e veículos como o TechRadar reportaram vulnerabilidades no MCP server oficial de Git mantido pela Anthropic. As falhas incluíam:

  • •Falta de validação de input que permitia path traversal
  • •Tool results que podiam conter prompt injection — o agente seguia instruções injetadas via dados do Git
  • •Ausência de rate limiting que permitia abuso da API

A lição: mesmo criadores do protocolo podem errar na implementação. Segurança precisa de revisão contínua.

Supply chain attacks via MCP

O ecossistema MCP enfrenta o mesmo risco que npm e PyPI: um atacante pode publicar um MCP server malicioso com nome similar a um popular (typosquatting), ou comprometer um server existente via dependência. Um server malicioso pode: ler arquivos sensíveis via resources, executar comandos arbitrários via tools, ou injetar prompts que redirecionam o comportamento do agente.

💡 Lições aprendidas

Sempre verifique a fonte de MCP servers antes de instalar. Prefira servers do Docker MCP Catalog (curados) ou do repositório oficial. Execute em sandbox quando possível. Monitore o tráfego de rede do server. E lembre: o modelo de ameaça para agentes é diferente — o atacante não precisa hackear seu sistema, basta convencer o LLM a agir em seu nome.

📜 Governança enterprise

Empresas grandes não podem simplesmente "deixar desenvolvedores usarem agentes CLI". Precisam de governança: controle de acesso, aprovações em cadeia, compliance frameworks e integração com infraestrutura de segurança existente.

Framework de governança para agentes CLI

RBAC para agentes

Roles definem o que cada nível de usuário pode fazer com agentes: dev (local only), lead (staging), ops (production com aprovação).

Aprovações em cadeia

Ações críticas requerem aprovação de múltiplas pessoas. Ex: agente quer modificar infra de produção → precisa de OK do dev + lead.

Compliance frameworks

SOC 2, HIPAA, GDPR exigem controles específicos. Audit trails, data retention, direito ao esquecimento aplicados a sessões de agentes.

Trust but verify

O padrão emergente: permitir o uso de agentes mas com logging completo, revisão periódica e alertas para anomalias.

A adoção enterprise de agentes CLI está seguindo um padrão familiar: começa com equipes early-adopter em modo "shadow IT", depois a empresa percebe que precisa de controle, e finalmente estabelece políticas formais. As empresas que anteciparem esse ciclo e criarem governança desde o início terão vantagem competitiva — menos incidentes, adoção mais rápida e confiança institucional no uso de agentes.
