Autenticação, permissões e controle de acesso

🔑 OAuth e tokens

Quando um agente CLI precisa acessar uma API externa (Google Cloud, GitHub, Slack), ele precisa se autenticar. O padrão dominante é OAuth 2.0, que permite ao usuário autorizar o agente sem compartilhar sua senha diretamente.

Fluxos OAuth para CLIs

Authorization Code Flow

O CLI abre o browser, o usuário faz login e autoriza. O browser redireciona para localhost com um code. O CLI troca o code por tokens. Mais seguro, usado pelo gws e gh CLI.

Device Flow

O CLI mostra um código e uma URL. O usuário vai ao browser, acessa a URL e digita o código. Ideal para ambientes sem browser local (SSH, containers). Usado pelo GitHub CLI em headless.

📊 API Keys vs OAuth Tokens

API Keys
  • - Simples: string fixa no header
  • - Não expira automaticamente
  • - Sem escopos granulares
  • - Risco: se vaza, acesso total
OAuth Tokens
  • - Complexo: fluxo de autorização
  • - Expira (refresh token renova)
  • - Escopos granulares por recurso
  • - Risco: limitado e revogável

👤 Identidade do agente vs do usuário

Quando o Claude Code faz um git push em seu nome, quem "fez" o push? Você ou o agente? Essa ambiguidade de identidade é um dos problemas mais complexos da era dos agentes CLI — e tem implicações legais, de compliance e de segurança.

O problema da identidade

💡 Solução emergente: identidades separadas

A tendência é criar identidades separadas para agentes, onde o token do agente carrega metadata indicando que é uma ação automatizada em nome de um humano específico. O GitHub já suporta isso com "bot accounts" e o MCP spec está adicionando campos de identidade no protocolo. Isso permite: auditoria precisa (saber que foi o agente), controle fino (limitar o que o agente pode fazer) e responsabilidade clara (o humano que autorizou).

🎫 Escopos e permissões

O princípio do menor privilégio é fundamental: o agente deve ter acesso apenas ao mínimo necessário para completar a tarefa. Escopos OAuth permitem limitar exatamente o que um token pode fazer, recurso por recurso.

Granularidade de escopos

Nível 1:
Read-only vs Read-write — A distinção mais básica. Um agente que analisa código precisa apenas de leitura; um que faz commits precisa de escrita.
Nível 2:
Por recurso — Acesso ao Calendar mas não ao Gmail. Acesso ao repo X mas não ao repo Y. Cada API define seus escopos.
Nível 3:
Por ação — Pode listar events mas não deletar. Pode criar branches mas não fazer merge em main. Granularidade máxima.

📊 Exemplo: gws com escopos Google

O gws (Google Workspace CLI) permite escolher escopos no momento da autenticação:

# Apenas Calendar (read-only)

gws auth --scopes calendar.readonly

# Calendar + Gmail (read-write)

gws auth --scopes calendar,gmail

# Tudo (perigoso!)

gws auth --scopes all

🔐 Secrets management

API keys, tokens e credenciais são os ativos mais sensíveis em qualquer sistema. A regra número um: nunca hardcode secrets no código. Agentes CLI precisam de secrets para funcionar, mas o acesso deve ser controlado e auditado.

✅ Boas práticas

  • •Env vars: export ANTHROPIC_API_KEY=sk-... — simples, funciona em CI
  • •Keychain do OS: macOS Keychain, GNOME Keyring — protegido por login
  • •Vaults: HashiCorp Vault, 1Password CLI — rotação e auditoria
  • •.gitignore: sempre incluir .env, *.key, credentials.json

❌ Más práticas

  • •Hardcode: API key direto no código-fonte ou em config versionado
  • •Shared secrets: mesma API key para todos da equipe
  • •Sem rotação: mesma key há anos sem nunca trocar
  • •Over-scoped: dar token com acesso total quando precisa só de leitura

⚠️ Agentes e secrets

Um risco específico dos agentes: se o agente tem acesso ao filesystem, ele pode ler o .env com todas as suas API keys. Configure o sandbox para que o agente acesse apenas os secrets que precisa. No Claude Code, nunca coloque secrets em CLAUDE.md — use env vars que o agente herda do shell.

5

✅ Aprovação humana

O modelo human-in-the-loop é a última linha de defesa: antes de executar ações críticas, o agente mostra o que pretende fazer e espera aprovação explícita do usuário. Cada agente CLI implementa isso de forma diferente.

Níveis de aprovação

AUTO

Safe actions — Leitura de arquivos, busca, git status. Executados automaticamente sem pedir permissão.

ASK

Risky actions — Edição de arquivos, execução de shell commands, git commit. O agente mostra o diff ou comando e pede OK.

BLOCK

Dangerous actions — rm -rf, deploy para produção, acesso a dados sensíveis. Bloqueados por padrão, mesmo com aprovação.

📊 Como cada agente implementa

Claude Code — Mostra diff completo antes de editar arquivos. Mostra comando completo antes de executar no shell. Aceita allowlists via settings.
Gemini CLI — Pede confirmação para shell commands. Sandbox mode executa tudo em ambiente isolado sem pedir aprovação.
Codex CLI — Roda tudo em sandbox Docker. Aplica mudanças ao host apenas com aprovação final (apply patch).
6

🏢 SSO e integração corporativa

Em empresas, a autenticação individual por OAuth não escala. A integração com SSO (Single Sign-On) via SAML ou OIDC permite que agentes CLI se autentiquem usando a identidade corporativa do usuário, com todos os controles centralizados.

Integração enterprise

SAML 2.0

Protocolo XML maduro. Usado por grandes empresas. O CLI redireciona para o IdP corporativo (Okta, Entra ID). Login único para todas as ferramentas.

OIDC (OpenID Connect)

Protocolo moderno baseado em OAuth 2.0. Retorna ID tokens com claims do usuário. Mais simples que SAML. Preferido por ferramentas cloud-native.

Provisioning (SCIM)

Criação e remoção automática de acesso quando funcionário entra/sai. O agente perde acesso no mesmo dia que o funcionário sai da empresa.

Conditional Access

Políticas que restringem acesso por IP, dispositivo, horário ou nível de risco. Agente em rede corporativa? OK. Agente em café? MFA obrigatório.

A integração de agentes CLI com infra enterprise existente é um dos maiores desafios de adoção. Empresas que já investiram em Active Directory/Entra ID, Okta ou Google Workspace esperam que agentes CLI respeitem as mesmas políticas de acesso que aplicações web. Os agentes que oferecerem SSO nativo e integração com IdPs corporativos terão vantagem na adoção enterprise.

Resumo Final

✓ OAuth 2.0 (authorization code e device flow) é o padrão para autenticação de agentes CLI
✓ A identidade do agente vs do usuário é um problema aberto — identidades separadas estão emergindo
✓ Escopos OAuth implementam o princípio do menor privilégio por recurso e ação
✓ Secrets nunca devem ser hardcoded — use env vars, keychains ou vaults
✓ Aprovação humana em três níveis: auto-approve safe, ask for risky, block dangerous
✓ SSO via SAML/OIDC e provisioning via SCIM integram agentes com infra corporativa
