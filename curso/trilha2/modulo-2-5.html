Saída estruturada: JSON, schemas e introspecção

Entenda por que JSON é a língua franca dos agentes, como schemas permitem auto-descoberta de APIs, e como CLIs inteligentes como o gws montam comandos dinamicamente a partir de descrições de serviço.

1

📊 Por que JSON

JSON se tornou a língua franca dos agentes de IA por uma razão simples: é parseável por máquinas e legível por humanos ao mesmo tempo. Quando um agente precisa processar a saída de um comando ou enviar dados para uma API, JSON é o formato que todo o ecossistema entende.

📊 Comparação de formatos

JSON Padrão dominante

Universal, nativo em browsers e APIs REST. Suportado por todos os LLMs. Parseável sem ambiguidade. Limitação: não suporta comentários.

YAML Config files

Mais legível para humanos. Suporta comentários. Padrão em Kubernetes, GitHub Actions, Docker Compose. Problemas: indentação sensível, parsing ambíguo.

XML Legacy

Verboso mas expressivo. Ainda usado em SOAP, SAML, Maven. Schemas poderosos (XSD). Raramente escolhido para novos projetos.

Protocol Buffers Performance

Binário, compacto, tipado. Usado internamente por Google (gRPC). Ótimo para comunicação machine-to-machine. Não legível por humanos.

💡 JSON e LLMs

LLMs são treinados com imensos volumes de JSON (APIs, configs, datasets). Isso significa que modelos como Claude, Gemini e GPT são extremamente bons em gerar e parsear JSON corretamente. Muitas APIs de LLM oferecem "structured output" mode que garante JSON válido na resposta. Essa sinergia entre formato e modelo é um dos motivos pelo qual JSON domina o ecossistema de agentes.

2

🔍 Introspecção de schema

APIs que se descrevem permitem que agentes descubram automaticamente quais operações estão disponíveis, quais parâmetros aceitar e quais formatos de resposta esperar. Isso elimina a necessidade de documentação manual e permite CLIs auto-atualizáveis.

Padrões de introspecção

OpenAPI / Swagger

Descreve REST APIs em JSON/YAML. Endpoints, parâmetros, schemas de request/response. Gera clientes automaticamente.

JSON Schema

Descreve a estrutura de documentos JSON. Tipos, validações, valores default. Usado pelo MCP para definir parâmetros de tools.

GraphQL Introspection

Query __schema retorna todo o tipo system. O client sabe exatamente o que pode pedir. Auto-documentação built-in.

Para agentes, a introspecção é fundamental. Quando um agente se conecta a um MCP server, a primeira coisa que faz é perguntar: "quais tools você tem e quais parâmetros elas aceitam?". O server responde com JSON Schema descrevendo cada tool. O LLM usa essa descrição para saber quando e como chamar cada tool — sem nenhuma documentação adicional.

📋 Discovery Service

O Google Discovery Service é um dos exemplos mais impressionantes de introspecção em produção. Ele descreve todas as APIs do Google como JSON schemas, e o gws usa isso para montar sua árvore de comandos dinamicamente em runtime — sem ter nenhum comando hardcoded.

📊 Poder do Discovery

O Google Discovery Service descreve mais de 300 APIs em um formato padronizado. Isso significa que o gws pode:

Auto-atualizar

Quando o Google adiciona um endpoint novo, o gws já suporta sem precisar de update

Zero hardcode

Nenhum comando é escrito manualmente — todos derivam do schema JSON

Validação automática

Parâmetros são validados contra o schema antes de enviar a request

Help dinâmico

Descrições de parâmetros e exemplos gerados a partir do schema

💡 Conceito para o futuro

O modelo do gws pode ser aplicado a qualquer API que tenha um schema. Imagine um CLI universal que lê OpenAPI specs e gera comandos para qualquer serviço REST. Combine isso com MCP e um agente de IA, e você tem um sistema que pode aprender a usar qualquer API automaticamente.

4

🔄 Dry-run e preview

O padrão --dry-run mostra o que um comando faria sem realmente executar. Para agentes, isso é essencial: permite simular ações antes de executá-las, reduzindo risco e aumentando a confiança na automação.

Dry-run em ação

$ gws calendar events insert --dry-run --calendar-id primary \

--summary "Sprint Review" --start "2026-03-25T14:00:00"

DRY RUN - Would execute:

POST https://www.googleapis.com/calendar/v3/calendars/primary/events

Body: {"summary": "Sprint Review", "start": {"dateTime": "2026-03-25T14:00:00"}}

No changes made.

✅ Quando usar dry-run

  • •Antes de operações destrutivas (delete, overwrite)
  • •Para validar parâmetros sem gastar quota da API
  • •Em pipelines de CI para preview de mudanças
  • •Quando o agente não tem certeza da ação correta

⚠️ Limitações

  • •Nem todos os CLIs suportam --dry-run
  • •Dry-run pode não capturar efeitos colaterais
  • •Estado pode mudar entre dry-run e execução real
  • •Não substitui sandboxing para segurança real
5

📄 Auto-paginação

APIs retornam dados paginados — em vez de enviar 10.000 resultados de uma vez, enviam 100 por página com um next_page_token. CLIs inteligentes lidam com isso automaticamente, mas precisam de cuidado para não consumir toda a cota da API.

Fluxo de auto-paginação

1 CLI faz request inicial → API retorna página 1 + nextPageToken
2 CLI detecta token → automaticamente faz request para página 2
3 Repete até: nextPageToken ausente, ou --max-results atingido, ou --pages limit
4 CLI combina todos os resultados e retorna como uma lista unificada

⚠️ O perigo da paginação sem limites

Um agente que pede "liste todos os emails" sem limite de paginação pode fazer centenas de requests, consumir toda a cota da API e gerar custos significativos. O gws resolve isso com --max-results para limitar a quantidade total e paginação automática inteligente que respeita rate limits. Sempre defina um teto quando delegar paginação a um agente.

6

🧪 Validação e testes

Como garantir que a saída de um CLI está sempre no formato esperado? JSON Schema validation e testes de contrato são as ferramentas que garantem que agentes podem confiar na saída estruturada das ferramentas que usam.

Camadas de validação

1. Schema Validation

Verificar se a saída JSON segue o schema declarado. Ferramentas: ajv (JavaScript), jsonschema (Python), zod (TypeScript). Roda em millisegundos, pega erros de tipo e campos faltantes.

2. Contract Testing

Garantir que o CLI sempre retorna o mesmo formato. "Se eu chamar gws calendar events list, o resultado sempre tem os campos id, summary, start?" Testes de contrato verificam isso em cada versão.

3. CI para CLIs

Rodar testes automatizados em cada mudança do CLI. Verificar que saída JSON não quebrou, que --dry-run funciona, que paginação respeita limites. GitHub Actions ou GitLab CI com suite de testes.

📊 Ferramentas de validação

ajv

JSON Schema validator mais rápido do ecossistema Node.js

zod

Schema declaration + validation em TypeScript com inferência de tipos

jest / vitest

Test runners com snapshot testing para saída JSON de CLIs

A validação é especialmente importante quando agentes encadeiam múltiplos CLIs: a saída de um vira entrada de outro. Se o formato muda silenciosamente, toda a cadeia quebra. Testes de contrato e CI contínuo são a rede de segurança que garante que a composição de ferramentas funciona de forma confiável.
