📋 Visão geral do pipeline

O pipeline CLI-Anything é uma sequência de 7 fases encadeadas, cada uma com input, processamento e output bem definidos. A execução é orquestrada por um agente LLM que toma decisões em cada etapa.

Fluxo completo do pipeline

O comando /cli-anything dispara o pipeline inteiro dentro de um agente (Claude Code, Gemini CLI etc). O agente segue um protocolo de fases:

F1

Clone

F2

Análise

F3

Design

F4

Impl.

F5

Testes

F6

Docs

F7

SKILL

📊 Métricas de execução típicas

Software simples (< 5K LOC): 10-15 minutos, ~50K tokens consumidos
Software médio (5-50K LOC): 20-35 minutos, ~150K tokens consumidos
Software complexo (> 50K LOC): 30-45 minutos, ~300K tokens consumidos

Cada fase produz artefatos intermediários que alimentam a fase seguinte. O agente pode pausar entre fases para solicitar feedback humano, mas o fluxo padrão é automático. A chave é que o agente raciocina sobre o código — não é execução mecânica, é análise inteligente guiada por um LLM.

2

🔍 Fase 1: Clone e preparação

A primeira fase é o reconhecimento do terreno: o agente clona o repositório e mapeia a estrutura do projeto, identificando linguagem, framework e arquitetura de diretórios.

Input → Processamento → Output

Input:

URL do repositório ou caminho local do projeto

Processamento:

  • 1. git clone do repositório
  • 2. Detecção de linguagem (package.json → Node.js, requirements.txt → Python, go.mod → Go)
  • 3. Identificação de framework (Express, Django, Flask, Spring...)
  • 4. Mapeamento de estrutura de diretórios (src/, lib/, api/, models/)
  • 5. Leitura de README e documentação existente

Output:

Project manifest — JSON com metadata do projeto: linguagem, framework, diretórios principais, entry points, dependências, scripts disponíveis

💡 O que o agente decide nesta fase

O agente não apenas lista arquivos — ele raciocina sobre o que encontra. Se vê um src/api/ e um src/ui/, já infere separação backend/frontend. Se encontra docker-compose.yml, entende que há serviços. Essa inteligência contextual é o que diferencia de ferramentas estáticas de análise.

3

🧠 Fase 2: Análise arquitetural

A fase mais intelectualmente intensa: o agente faz análise estática profunda do código, identificando módulos, classes, funções, dependências e fluxos de dados para mapear funcionalidades.

O que o agente examina

Módulos e classes

Quais existem, como se relacionam, qual a hierarquia

Funções exportadas

Quais funções são públicas e podem virar comandos

Dependências e imports

Grafo de dependências entre módulos

Fluxos de dados

Como dados fluem do input ao output

Pontos de entrada

REST endpoints, event handlers, rotas

Modelo de dados

Schemas, entidades, tipos de dados

O output desta fase é um architecture map — um documento estruturado que lista todas as funcionalidades identificadas, classificadas por domínio (usuarios, tarefas, configurações), com referência aos arquivos e funções onde estão implementadas. Este mapa é o insumo fundamental para a fase de design.

📊 Exemplo de architecture map

{
  "domains": [
    {
      "name": "users",
      "source_files": ["src/models/user.js", "src/api/users.js"],
      "capabilities": ["create", "list", "update", "delete", "authenticate"],
      "entry_points": ["POST /api/users", "GET /api/users", "PUT /api/users/:id"]
    },
    {
      "name": "tasks",
      "source_files": ["src/models/task.js", "src/api/tasks.js"],
      "capabilities": ["create", "list", "update", "delete", "assign", "filter"],
      "entry_points": ["POST /api/tasks", "GET /api/tasks"]
    }
  ]
}
4

✏️ Fase 3: Design do CLI

Onde a análise vira design: o agente traduz as funcionalidades mapeadas em uma árvore de comandos CLI com nomenclatura intuitiva, flags consistentes e tipos validados.

Princípios de design aplicados

O agente segue princípios de design de CLI consagrados para gerar comandos ergonômicos:

  • 1.Hierarquia: app [recurso] [ação] [flags] — ex: myapp users create --name "João"
  • 2.Verbos para ações: create, list, get, update, delete, export, import
  • 3.Substantivos para recursos: users, tasks, projects, settings
  • 4.Flags consistentes: --output json|table|csv, --verbose, --dry-run em todos os comandos
  • 5.Defaults sãos: output humano por padrão, JSON via flag, verbose desligado

O output desta fase é uma CLI spec — um JSON schema que define cada comando, seus argumentos, flags, tipos, valores default e exemplos de uso. Esta spec é o contrato que a fase de implementação vai seguir à risca.

💡 Por que o design é uma fase separada

Separar design de implementação é intencional. Se o agente fosse direto do código para a implementação, o resultado seria uma CLI que espelha a arquitetura interna (confusa) do software. A fase de design permite repensar a interface do ponto de vista do usuário/agente, não do desenvolvedor original. O resultado é uma CLI mais limpa e intuitiva.

5

🔧 Fase 4: Implementação

A fase onde código é gerado: o agente implementa a CLI completa — módulos, handlers, state management e REPL — com integração real ao software original.

Componentes gerados

1.
Entry point (bin/cli.js ou equivalente): Parser de argumentos, roteamento para subcomandos, inicialização
2.
Command handlers: Um módulo por domínio (users.js, tasks.js), cada um com funções para cada ação
3.
State manager: Persistência de sessão, tokens de auth, contexto entre comandos
4.
Output formatter: Conversão entre JSON (para agentes) e tabela/texto (para humanos)
5.
REPL (Read-Eval-Print Loop): Modo interativo com auto-complete, histórico e estado persistente

🚨 O ponto crítico: integração real vs mock

A implementação pode seguir três caminhos dependendo da arquitetura do software-alvo: (1) Se tem API REST/GraphQL, o CLI chama os endpoints diretamente. (2) Se tem biblioteca/módulo exportado, o CLI importa e chama funções. (3) Se é monolítico sem separação, o CLI pode gerar mocks que simulam funcionalidade sem controlar o software real. Sempre verifique qual caminho foi tomado.

6

🧪 Fases 5-7: Testes, documentação e SKILL.md

As três fases finais completam o pacote: testes automatizados, documentação abrangente e o arquivo SKILL.md que torna o CLI operável por outros agentes LLM.

As 3 fases finais

Fase 5 — Testes automatizados

Testes unitários para cada comando: input válido → output esperado, input inválido → erro correto, edge cases. Testes E2E para fluxos completos (criar → listar → editar → deletar). Framework escolhido conforme a linguagem do CLI.

Fase 6 — Documentação

README.md com instalação, uso básico e exemplos. Flag --help implementada para cada comando e subcomando. Exemplos práticos que agentes podem copiar diretamente.

Fase 7 — SKILL.md

O arquivo que transforma o CLI em tool agent-native. Contém: nome da ferramenta, descrição, lista de comandos com exemplos, limitações conhecidas. LLMs leem este arquivo para saber como operar o CLI sem supervisão.

SKILL.md — O contrato para agentes

O SKILL.md é talvez o artefato mais inovador do pipeline. Ele funciona como uma "API documentation para agentes LLM":

# myapp CLI

## Description
CLI for managing tasks and users in MyApp.

## Commands
- `myapp users list [--format json]` — List all users
- `myapp users create --name NAME --email EMAIL` — Create user
- `myapp tasks create --title TITLE --assignee USER_ID` — Create task
- `myapp tasks list [--status open|closed] [--format json]`

## Examples
```bash
myapp users create --name "João" --email "j@x.com" --format json
myapp tasks list --status open --format json
```

## Limitations
- Requires auth: run `myapp auth login` first
- Max 100 results per list query

Resumo Final

✓ O pipeline tem 7 fases sequenciais: Clone → Análise → Design → Implementação → Testes → Docs → SKILL.md
✓ Fase 1 gera o project manifest; Fase 2 produz o architecture map com funcionalidades mapeadas
✓ Fase 3 separa design de implementação para criar uma CLI centrada no usuário, não no código interno
✓ Fase 4 gera código real: handlers, state, REPL, formatters — com risco de mock em softwares monolíticos
✓ O SKILL.md (Fase 7) é o diferencial: transforma o CLI em ferramenta operável por qualquer agente LLM
