Análise e Design: entendendo o software-alvo

Como o pipeline examina código-fonte por meio de análise estática, mapeia funcionalidades em ações CLI e projeta schemas de comandos ergonômicos e consistentes.

1

🔬 Análise estática de código

O agente LLM não executa o software — ele lê e interpreta o código-fonte estaticamente, usando sua capacidade de compreender linguagens de programação para mapear a arquitetura interna.

O que o agente examina

A análise estática do pipeline opera sobre múltiplas camadas do código simultaneamente:

AST (Abstract Syntax Tree)

Estrutura sintática: funções, classes, variáveis, tipos

Imports e exports

O que cada módulo expõe e o que consome

Grafos de chamada

Qual função chama qual, profundidade de encadeamento

Herança e composição

Relações entre classes e interfaces

A grande vantagem de usar um LLM para análise estática (em vez de ferramentas tradicionais como ESLint, Pylint ou SonarQube) é que o LLM entende semântica, não apenas sintaxe. Ele sabe que uma função chamada createNewProject() provavelmente cria um projeto, e que handleDeleteRequest() trata de exclusão — mesmo sem annotations ou tipos explícitos.

💡 A diferença do LLM

Ferramentas tradicionais de análise estática produzem grafos de dependência e métricas de complexidade. Um LLM produz compreensão: ele lê o código como um desenvolvedor sênior faria, infere propósitos, identifica padrões arquiteturais (MVC, Repository, Service Layer) e raciocina sobre o que deveria virar comando CLI.

2

🗺️ Mapeamento de funcionalidades

A fase de mapeamento traduz elementos de GUI em ações potenciais: cada botão, menu e formulário no software original corresponde a uma função no código que pode virar um comando CLI.

Processo de mapeamento

1. Identificar ações na GUI

O agente analisa event handlers, formulários, botões submit, links de navegação no código frontend. Cada um é uma "ação potencial".

2. Rastrear até a função backend

Cada ação GUI geralmente chama uma API ou função no backend. O agente rastreia: onClick → fetch('/api/users') → usersController.create().

3. Extrair parâmetros

Os campos do formulário viram argumentos do comando: input "Nome" → --name, select "Role" → --role admin|user, checkbox "Ativo" → --active.

4. Mapear para comando CLI

Resultado: "Botão Criar Usuário" → app users create --name "X" --role admin

🚨 Desafio: ações sem equivalente CLI limpo

Nem toda ação GUI tem tradução direta. Exemplos problemáticos: drag-and-drop para reordenar itens, draw em canvas, gestos de pinch-to-zoom, wizards multi-step com lógica condicional. O pipeline precisa decidir: simplificar (reordenar vira app items reorder --id X --position 3) ou omitir.

3

📐 Princípios de design de CLI

O design de uma boa CLI segue princípios consagrados. O pipeline aplica essas regras para gerar interfaces intuitivas, consistentes e ergonômicas — tanto para humanos quanto para agentes.

Regras de design

Nomenclatura: Verbos para ações (create, delete, list, get, update, export), substantivos para recursos (users, tasks, projects).
Hierarquia: app [recurso] [ação] [flags] — inspirado em kubectl, gh, az. Permite auto-complete e descoberta progressiva.
Consistência: Mesmas flags para mesmos conceitos em todos os comandos: --output, --format, --verbose, --quiet, --dry-run.
Ergonomia: Defaults sãos (output humano, verbose off), help integrado, mensagens de erro claras com sugestão de correção.

✅ Bom design

  • app users create --name "João"
  • app tasks list --status open
  • app projects export --format csv
  • Previsível, consistente, auto-explicativo

❌ Design ruim

  • app newUser João
  • app getTasks -s open
  • app dump projects csv
  • Inconsistente, abreviações, sem padrão
4

🔗 Identificação de APIs internas

O pipeline não depende de APIs públicas documentadas — ele procura qualquer ponto de entrada no código que possa ser invocado programaticamente, incluindo APIs internas nunca expostas ao público.

O que o pipeline procura

REST endpoints

app.get(), app.post(), @GetMapping, router.handle()

GraphQL resolvers

Query e Mutation resolvers com schemas tipados

Funções exportadas

module.exports, export function, pub fn

Event handlers

on('event'), @EventListener, subscriber patterns

Quando o software tem API interna (mesmo que não documentada), o CLI gerado chama essas funções diretamente. Quando não tem nenhum ponto de entrada limpo, o pipeline tenta extrair a lógica de negócio das funções embutidas na UI e refatorá-las em chamadas isoladas. Este é o cenário mais desafiador e o que mais se beneficia do refinamento iterativo.

📊 Taxa de sucesso por tipo de arquitetura

API REST clara: ~95% das funcionalidades mapeáveis com integração real
Biblioteca com exports: ~85% mapeável — depende da qualidade da abstração
Monolítico sem separação: ~40-60% — muitas funcionalidades viram mock

📊 Schema de comandos

Cada comando do CLI gerado é definido por um JSON schema formal que especifica nome, descrição, argumentos, flags, tipos e exemplos. Este schema alimenta o REPL, o --help e o SKILL.md.

Estrutura de um command schema

{
  "command": "users create",
  "description": "Create a new user account",
  "arguments": [
    {
      "name": "name",
      "type": "string",
      "required": true,
      "description": "Full name of the user"
    }
  ],
  "flags": [
    {
      "name": "--email",
      "type": "string",
      "required": true,
      "description": "User email address"
    },
    {
      "name": "--role",
      "type": "enum",
      "values": ["admin", "user", "viewer"],
      "default": "user",
      "description": "User role"
    },
    {
      "name": "--format",
      "type": "enum",
      "values": ["json", "table", "csv"],
      "default": "table",
      "description": "Output format"
    }
  ],
  "examples": [
    "myapp users create \"João\" --email j@x.com",
    "myapp users create \"Ana\" --email a@x.com --role admin --format json"
  ]
}

💡 Validação via schema

O schema não é apenas documentação — ele é usado em runtime para validar inputs. Se um usuário ou agente passa --role superadmin e o enum só aceita admin|user|viewer, o CLI rejeita com uma mensagem clara. Isso evita erros silenciosos e torna a CLI previsível para agentes automatizados.

6

🎯 Priorização de funcionalidades

Nem toda funcionalidade do software precisa virar comando CLI. O pipeline aplica critérios de priorização para focar no que realmente importa para automação e agentes.

✅ Priorizar

  • •CRUD operations: criar, ler, atualizar, deletar — core de qualquer sistema
  • •Queries e filtros: listar com filtros, buscar por critérios
  • •Exports: exportar dados em JSON, CSV, PDF
  • •Auth e config: login, logout, configurações
  • •Ações batch: operações em lote que se beneficiam de automação

⚠️ Deprioritizar

  • •Flows visuais complexos: wizards multi-step, flows condicionais
  • •Drag-and-drop: reordenamento visual, canvas interativo
  • •Configurações de layout: temas, posicionamento visual, UI prefs
  • •Real-time visual: dashboards live, gráficos interativos
  • •Interações modais: confirmações que dependem de contexto visual

💡 A regra 80/20

Na prática, 80% do valor de automação vem de 20% das funcionalidades: CRUD + queries + exports. Se o CLI gerado cobre bem essas operações, já resolve a maioria dos casos de uso para agentes. O refinamento de funcionalidades visuais complexas pode vir depois, se necessário.

Resumo Final

✓ Análise estática com LLM vai além da sintaxe — entende semântica, propósitos e padrões arquiteturais
✓ O mapeamento GUI → CLI rastreia ações visuais até funções backend e extrai parâmetros como flags
✓ Design de CLI segue princípios: hierarquia recurso/ação, verbos consistentes, flags padronizadas
✓ APIs internas (REST, GraphQL, exports) são os melhores pontos de integração — monolíticos geram mocks
✓ Priorizar CRUD, queries e exports (80% do valor) sobre funcionalidades visuais complexas
