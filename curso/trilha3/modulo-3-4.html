Implementação: estado, REPL e saída JSON

Como o pipeline gera CLIs com gerenciamento de estado persistente, REPL interativo, saída JSON estruturada e integração real com o software original.

Intermediário-Avançado Nível
Técnico Tipo
1

💾 Gerenciamento de estado

CLIs tradicionais são stateless — cada comando é independente. CLI-Anything gera CLIs stateful: login persistente, sessão mantida, contexto acumulado entre comandos.

Camadas de estado

O state management da CLI gerada opera em três camadas, do mais efêmero ao mais persistente:

L1
Sessão (memória): Contexto do REPL atual — último recurso acessado, resultados em cache, variáveis temporárias. Perdido ao fechar o REPL.
L2
Arquivo local (~/.myapp/state.json): Tokens de auth, preferências, projeto ativo, último workspace. Persiste entre sessões.
L3
Keychain/credential store: Tokens sensíveis armazenados no sistema operacional (macOS Keychain, Linux secret-service). Máxima segurança.

O estado é crucial para agentes: sem estado, cada comando precisaria re-autenticar, re-selecionar o projeto, re-configurar preferências. Com estado, o agente faz myapp auth login uma vez e todos os comandos subsequentes herdam a sessão. Isso reduz o número de tokens gastos e torna fluxos complexos viáveis.

💡 Estado para agentes vs humanos

Humanos geralmente configuram estado uma vez e esquecem. Agentes precisam de verificação explícita: antes de executar um comando, o agente ideal checa myapp status para confirmar que está autenticado e no projeto correto. O CLI gerado inclui esse comando de verificação por padrão.

🔄 REPL interativo

O REPL (Read-Eval-Print Loop) é um modo interativo onde o usuário ou agente pode executar comandos sequenciais sem reiniciar a CLI, mantendo estado e contexto entre cada operação.

Anatomia do REPL gerado

$ myapp repl
Welcome to MyApp CLI v1.0 (type 'help' for commands)

myapp> auth login --email j@x.com
✓ Authenticated as João (token saved)

myapp> users list --format table
┌────┬──────────┬───────────────┬────────┐
│ ID │ Name     │ Email         │ Role   │
├────┼──────────┼───────────────┼────────┤
│ 1  │ João     │ j@x.com       │ admin  │
│ 2  │ Ana      │ a@x.com       │ user   │
└────┴──────────┴───────────────┴────────┘

myapp> tasks create --title "Review PR" --assignee 2
✓ Task #42 created (assigned to Ana)

myapp> exit
Goodbye!
Auto-complete

Tab completa comandos, subcomandos, flags e valores de enum

Histórico

Setas up/down navegam histórico, Ctrl+R busca

Estado persistido

Auth, contexto e resultados anteriores mantidos

Variáveis de sessão

$last_id, $last_result para encadear comandos

📊 REPL vs comandos avulsos

Comando avulso: myapp users list — stateless, cada execução é independente
Modo REPL: myapp repl → sessão contínua com estado, ideal para agentes executando múltiplas operações
Agentes tipicamente usam: comandos avulsos com --format json, mas REPL quando precisam de sessão stateful

📊 Saída JSON estruturada

Toda saída da CLI gerada opera em dual mode: human-readable (formatado, colorido, tabelas) e machine-readable (JSON puro). Agentes sempre usam o modo JSON.

Estrutura padronizada de output JSON

{
  "status": "success",
  "data": {
    "users": [
      { "id": 1, "name": "João", "email": "j@x.com", "role": "admin" },
      { "id": 2, "name": "Ana", "email": "a@x.com", "role": "user" }
    ]
  },
  "metadata": {
    "total": 2,
    "page": 1,
    "per_page": 50,
    "command": "users list",
    "timestamp": "2026-03-23T10:30:00Z"
  },
  "errors": []
}

A estrutura é sempre a mesma: status (success|error), data (payload principal), metadata (paginação, timestamps, comando executado) e errors (array de erros com código e mensagem). Essa consistência permite que qualquer agente parse o output sem conhecer previamente o schema específico de cada comando.

✅ Modo humano (default)

$ myapp users list
┌────┬──────┬──────────┐
│ ID │ Name │ Role     │
├────┼──────┼──────────┤
│ 1  │ João │ admin    │
│ 2  │ Ana  │ user     │
└────┴──────┴──────────┘
2 users found

🤖 Modo JSON (agentes)

$ myapp users list --json
{"status":"success",
 "data":{"users":[
   {"id":1,"name":"João"},
   {"id":2,"name":"Ana"}
 ]},
 "metadata":{"total":2}}

🔌 Módulos e subcomandos

O CLI gerado é organizado em módulos — cada módulo agrupa comandos relacionados a um domínio, com lazy loading para performance.

Estrutura de módulos típica

myapp-cli/
├── bin/
│   └── myapp.js          # Entry point
├── src/
│   ├── commands/
│   │   ├── auth.js        # myapp auth login|logout|status
│   │   ├── users.js       # myapp users create|list|get|update|delete
│   │   ├── tasks.js       # myapp tasks create|list|assign|complete
│   │   └── config.js      # myapp config set|get|list
│   ├── lib/
│   │   ├── state.js       # State management (L1, L2, L3)
│   │   ├── output.js      # JSON/table/csv formatter
│   │   └── client.js      # HTTP client for API calls
│   └── repl.js            # REPL mode
├── SKILL.md
└── package.json

Cada módulo é carregado sob demanda (lazy loading): quando o usuário executa myapp users list, apenas o módulo users.js é importado. Isso mantém o tempo de startup baixo mesmo em CLIs com dezenas de módulos — importante para agentes que executam centenas de comandos por sessão.

💡 Extensibilidade

A estrutura modular permite que usuários adicionem novos módulos sem modificar o core da CLI. Basta criar um novo arquivo em src/commands/ seguindo o padrão e registrá-lo. Isso facilita iterações de refinamento: se o pipeline não gerou um comando necessário, é simples adicioná-lo manualmente.

⚙️ Configuração e flags universais

O pipeline gera um conjunto de flags universais presentes em todos os comandos, além de flags específicas extraídas dos parâmetros de cada função original.

Flags universais geradas automaticamente

--json Output em JSON puro (equivalente a --format json). Atalho para agentes.
--format Formato de saída: json | table | csv. Default: table para humanos.
--dry-run Simula a operação sem executar. Mostra o que seria feito. Essencial para testes.
--verbose Modo debug: mostra requests HTTP, queries SQL, timing de operações.
--quiet Suprime outputs não-essenciais. Apenas data ou erros. Para pipelines.
--config Caminho para arquivo de configuração alternativo (~/.myapp/config.json).

📊 Flags específicas por comando

Além das flags universais, cada comando recebe flags extraídas dos parâmetros da função original:

Parâmetro name: string → flag --name STRING
Parâmetro isActive: boolean → flag --active / --no-active
Parâmetro status: "open" | "closed" → flag --status open|closed
Parâmetro limit: number = 50 → flag --limit NUMBER (default: 50)

🔗 Integração com o software original

O ponto mais crítico de toda a implementação: o CLI gerado controla o software real ou é apenas uma simulação? A resposta depende da arquitetura do software-alvo.

3 cenários de integração

Cenário 1: Software com API (melhor caso)

O CLI chama os endpoints REST/GraphQL diretamente. Exemplo: fetch("http://localhost:3000/api/users"). Integração real, resultados reais, controle total. O CLI é um client HTTP tipado.

Cenário 2: Software como biblioteca (bom caso)

O CLI importa módulos do software e chama funções diretamente. Exemplo: const { createUser } = require('../src/services/users'). Integração real, mas requer que o software suporte ser importado como lib.

Cenário 3: Software monolítico (caso problemático)

Não há API nem módulos importáveis. O pipeline pode gerar mocks que simulam funcionalidade: salvar em arquivo local, manter estado próprio, mas sem controlar o software real. O CLI "funciona" mas não está conectado.

🚨 Como detectar mock vs real

  • •Verifique o código gerado: procure por fetch(), require() ou import apontando para o software original
  • •Teste cruzado: crie algo via CLI e verifique se aparece na GUI do software (e vice-versa)
  • •Procure por JSON/SQLite local: se o CLI armazena dados em arquivo próprio em vez de chamar o software, é mock
  • •Flag de verificação: CLIs bem gerados incluem myapp doctor que verifica a conexão com o software

Resumo Final

✓ CLIs geradas são stateful: sessão em memória, arquivo local, keychain — estado persiste entre comandos
✓ REPL interativo com auto-complete, histórico e variáveis de sessão para fluxos complexos
✓ Saída dual-mode: humano (tabelas coloridas) e máquina (JSON padronizado com status/data/metadata/errors)
✓ Flags universais (--json, --dry-run, --verbose, --quiet) + flags específicas extraídas das funções originais
✓ Integração real depende da arquitetura: API = melhor caso, monolítico = risco de mock — sempre verifique
