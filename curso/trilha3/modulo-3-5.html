Testes, documentação e SKILL.md

🧪 Testes unitários gerados

O pipeline gera automaticamente testes unitários para cada comando da CLI: input válido, input inválido e edge cases. A cobertura alvo é de pelo menos 3 cenários por comando.

Estrutura de testes por comando

describe('users create', () => {
  test('creates user with valid input', async () => {
    const result = await cli('users create --name "João" --email j@x.com');
    expect(result.status).toBe('success');
    expect(result.data.user.name).toBe('João');
    expect(result.data.user.id).toBeDefined();
  });

  test('rejects missing required field', async () => {
    const result = await cli('users create --name "João"');
    expect(result.status).toBe('error');
    expect(result.errors[0].code).toBe('MISSING_REQUIRED');
    expect(result.errors[0].field).toBe('email');
  });

  test('rejects invalid email format', async () => {
    const result = await cli('users create --name "João" --email not-email');
    expect(result.status).toBe('error');
    expect(result.errors[0].code).toBe('INVALID_FORMAT');
  });

  test('handles duplicate email gracefully', async () => {
    await cli('users create --name "João" --email j@x.com');
    const result = await cli('users create --name "Ana" --email j@x.com');
    expect(result.status).toBe('error');
    expect(result.errors[0].code).toBe('DUPLICATE');
  });
});

📊 Métricas de cobertura típicas

Comandos CRUD: 4-6 testes cada (create, read, update, delete + edge cases)
Comandos de query: 3-5 testes cada (filtros, paginação, formato de saída)
Auth: 4-6 testes (login success, login fail, token expired, logout)
Flags universais: 2-3 testes (--json valida schema, --dry-run não muta, --verbose mostra debug)

🔄 Testes E2E — fluxos completos

Além dos testes unitários, o pipeline gera testes end-to-end que exercitam fluxos completos: criar → listar → editar → deletar, com verificação de estado e saída JSON.

Tipos de testes E2E

Fluxo CRUD completo

create → list (verify exists) → update → get (verify change) → delete → list (verify gone)

Testes de REPL

Sequência de comandos em sessão REPL com verificação de estado entre cada um

Testes de estado

Login → execute command → logout → verify command fails without auth

Testes de output JSON

Validação do JSON contra schema: todos os campos obrigatórios, tipos corretos, estrutura padronizada

Testes E2E são particularmente importantes para CLI-Anything porque verificam se a integração com o software original é real e funcional. Se o teste E2E de "criar usuário → listar → verificar que existe" falha, isso indica que o CLI pode estar operando como mock, não controlando o software de verdade.

💡 Testes como detector de mocks

O teste E2E mais revelador é o teste cruzado: criar algo via CLI e verificar na GUI (ou no banco de dados) se realmente foi criado. Se o pipeline gerou uma CLI mock, esse teste falha imediatamente. Incluir testes cruzados no refinamento é a melhor forma de garantir integração real.

📝 Documentação automática

A Fase 6 gera documentação completa: README.md, --help para cada comando e exemplos práticos que servem tanto para humanos quanto como referência para agentes.

Artefatos de documentação

README.md

Instalação (npm install -g, pip install, binary download), setup inicial (auth, config), quick start com 5-10 comandos essenciais, link para docs completas.

--help integrado

Cada comando e subcomando responde a --help com: descrição, argumentos, flags, exemplos de uso. Gerado a partir do command schema (Fase 3).

Exemplos práticos

Cenários reais documentados: "Como criar um projeto do zero", "Como exportar relatório mensal", "Como configurar integração com CI/CD". Copy-pasteable.

Changelog (em iterações)

Quando o pipeline roda múltiplas iterações de refinamento, gera changelog documentando comandos adicionados, bugs corrigidos, outputs melhorados.

Exemplo de --help gerado

$ myapp users create --help

Create a new user account in the system.

Arguments:
  name                    Full name of the user (required)

Options:
  --email <email>         Email address (required)

🤖 SKILL.md — O contrato para agentes LLM

O SKILL.md é o artefato mais inovador do pipeline — funciona como "documentação de API para agentes LLM", permitindo que qualquer agente saiba operar a CLI sem supervisão humana.

Estrutura do SKILL.md

# MyApp CLI Tool

## Description
Command-line interface for managing projects, users and tasks
in MyApp. Supports CRUD operations, queries with filters,
data export, and interactive REPL mode.

## Prerequisites
- MyApp server running on localhost:3000
- Authenticated session (run `myapp auth login` first)

## Commands

### Authentication
- `myapp auth login --email EMAIL --password PASS`
- `myapp auth logout`
- `myapp auth status` — Check current auth state

### Users
- `myapp users list [--role ROLE] [--format json|table]`
- `myapp users create --name NAME --email EMAIL [--role ROLE]`
- `myapp users get ID [--format json]`
- `myapp users update ID [--name NAME] [--role ROLE]`
- `myapp users delete ID [--force]`

### Tasks
- `myapp tasks list [--status open|closed] [--assignee ID]`
- `myapp tasks create --title TITLE [--assignee ID] [--priority 1-5]`
- `myapp tasks complete ID`

## Output Format
All commands support `--json` flag for machine-readable output.
JSON structure: { status, data, metadata, errors }

## Limitations
- Max 100 results per list query (use --page for pagination)
- File uploads not supported via CLI
- Real-time notifications not available (poll with list commands)

## Error Handling
Errors return JSON with status:"error" and errors array.
Common codes: AUTH_REQUIRED, NOT_FOUND, VALIDATION_ERROR, DUPLICATE.

💡 Por que o SKILL.md muda tudo

Sem SKILL.md, um agente LLM precisa ler toda a documentação, tentar comandos por trial-and-error, e pode cometer erros. Com SKILL.md, o agente sabe exatamente quais comandos existem, quais parâmetros são obrigatórios, qual o formato de saída e quais as limitações. A taxa de sucesso de agentes operando CLIs com SKILL.md é significativamente maior que sem ele.

📊 SKILL.md como padrão emergente

Claude Code: lê SKILL.md automaticamente quando está na raiz do projeto ou em .claude/
Gemini CLI: extensões com SKILL.md são registradas como tools disponíveis
MCP: SKILL.md pode ser convertido em tool definitions para Model Context Protocol

📋 Manifest e metadata

O pacote gerado inclui metadata estruturada (package.json, pyproject.toml ou equivalente) que facilita instalação, descoberta e integração com catálogos de ferramentas.

Metadata gerada para catálogos

{
  "name": "myapp-cli",
  "version": "1.0.0",
  "description": "CLI for MyApp - manage projects, users and tasks",
  "bin": { "myapp": "./bin/myapp.js" },
  "keywords": ["cli", "myapp", "agent-native", "cli-anything"],
  "engines": { "node": ">=18" },
  "cli-anything": {
    "source": "https://github.com/org/myapp",
    "generated": "2026-03-23",
    "pipeline_version": "1.2.0",
    "agent_compatibility": ["claude-code", "gemini-cli", "codex-cli"],
    "integration_type": "api",
    "commands_count": 15,
    "has_repl": true,
    "has_skill_md": true
  }
}

A seção cli-anything no manifest é metadata específica que catálogos de ferramentas usam para indexar o CLI: qual software foi agentificado, qual tipo de integração (API, biblioteca, mock), compatibilidade com agentes, e se tem SKILL.md. Isso facilita a descoberta automática por agentes procurando ferramentas.

🔍 Refinamento iterativo

O primeiro resultado do pipeline raramente é perfeito. O refinamento iterativo é parte fundamental do processo — tipicamente 2-3 ciclos são necessários para um CLI robusto.

Ciclo de refinamento

1. Gerar (pipeline completo)

Executar as 7 fases pela primeira vez. Output: CLI v0.1 com comandos básicos.

2. Testar (manual + automatizado)

Rodar os testes gerados. Testar manualmente os comandos principais. Verificar integração real vs mock.

3. Identificar gaps

Comandos faltantes, outputs mal formatados, estado inconsistente, integração incompleta, erros de tipo.

4. Refinar (feedback ao agente)

Dar feedback específico ao agente: "o comando tasks list não retorna o campo assignee", "auth login não persiste o token". Re-executar fases relevantes.

5. Repetir (2-3x)

Cada iteração melhora significativamente o resultado. Na terceira iteração, a maioria dos CLIs está funcional e robusta.

🚨 Áreas comuns que precisam de refinamento

  • •Comandos faltantes: funcionalidades importantes que o pipeline não mapeou
  • •Outputs incompletos: campos faltando no JSON, formatação de tabela quebrada
  • •Estado inconsistente: auth que não persiste, contexto perdido entre comandos
  • •Integração parcial: alguns comandos reais, outros mock — precisa unificar
  • •SKILL.md desatualizado: não reflete os comandos que realmente foram implementados

💡 Técnica avançada

A melhor forma de testar refinamento é usar o CLI como se fosse um agente: pedir a outro agente LLM para executar uma tarefa usando apenas o SKILL.md e os comandos disponíveis. Onde o agente falha = onde o CLI precisa de refinamento. Esse "agente como QA" é a forma mais eficiente de encontrar gaps.

Resumo Final

✓ Testes unitários cobrem input válido, inválido e edge cases — pelo menos 3 cenários por comando
✓ Testes E2E verificam fluxos completos e são o melhor detector de integração mock vs real
✓ Documentação inclui README, --help por comando e exemplos práticos copy-pasteable
✓ SKILL.md é o contrato para agentes LLM — permite operação autônoma do CLI sem trial-and-error
✓ Refinamento iterativo (2-3 ciclos) é essencial — usar "agente como QA" é a melhor técnica
