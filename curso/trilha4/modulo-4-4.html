Todos os serviços GWS

A inovação do gws: em vez de codificar cada endpoint de cada API manualmente, ele gera comandos dinamicamente a partir dos JSON schemas das APIs do Google. Isso significa que quando o Google adiciona um novo endpoint, o gws já suporta automaticamente — sem atualização de código.

💡 Por que isso é revolucionário

CLIs tradicionais são escritos à mão: cada comando, cada flag, cada validação. O gws inverte essa lógica — os comandos são derivados automaticamente da especificação da API. É CLI-as-code-generation aplicado ao ecossistema Google inteiro.

2

🔑 Autenticação

OAuth 2.0 com Google Cloud Project. Setup único que abre acesso a todos os serviços do Workspace via terminal.

Setup de autenticação

1. Criar Google Cloud Project

No Google Cloud Console, crie um projeto e habilite as APIs do Workspace que deseja usar (Gmail, Drive, Calendar, etc.).

2. Configurar OAuth consent screen

Defina o nome do app, escopos necessários e usuários de teste. Em modo testing, apenas usuários explicitamente listados podem autenticar.

3. Criar credenciais OAuth

Gere um Client ID e Client Secret do tipo "Desktop Application". Baixe o JSON de credenciais.

4. Autenticar via gws

Execute gws auth login — abre o navegador para OAuth, salva tokens localmente.

⚠️ Limitação importante

Apps não verificados pelo Google operam em "modo testing" com limite de 100 usuários de teste. Para uso em produção, é necessário passar pelo processo de verificação do Google — que pode levar semanas. Para uso pessoal e de time pequeno, o modo testing é suficiente.

🏗️ Discovery Service

A inovação central do gws: em vez de CLI estático, comandos são gerados em runtime a partir dos JSON schemas das APIs. Sempre atualizado. Sempre completo.

Comandos prontos para uso

O Google Discovery Service é uma meta-API que descreve todas as outras APIs do Google em formato JSON. O gws lê essas descrições e monta os comandos automaticamente: verbos, parâmetros, tipos, validação — tudo derivado do schema. Novo endpoint na API = novo comando no CLI automaticamente, sem deploy de código.

📊 CLI estático vs dinâmico

CLI estático: comandos codificados manualmente — exige atualização a cada mudança na API
CLI dinâmico (gws): comandos gerados do schema — atualiza automaticamente com a API
Implicação: gws cobre 100% das APIs do Workspace sem código manual para cada endpoint

📊 Saída JSON e --dry-run

Toda saída em JSON estruturado. Ideal para agentes que precisam de dados parseáveis, simulação sem execução e paginação automática.

✅ Para agentes

  • •JSON nativo: toda saída é JSON parseável — sem scraping de texto
  • •--dry-run: simula o comando sem executar — agente pode validar antes de agir
  • •Auto-paginação: resultados grandes são paginados automaticamente
  • •Introspecção: o agente pode consultar o schema dos comandos em runtime

⚠️ CLIs tradicionais

  • •Saída textual: formatada para humanos, difícil de parsear
  • •Sem dry-run: tem que executar para ver o que acontece
  • •Paginação manual: --page 1, --page 2, --page 3...
  • •Sem introspecção: agente precisa ler docs externas

O --dry-run é especialmente valioso para agentes: antes de enviar um email ou deletar um arquivo, o agente pode simular a operação e mostrar ao usuário o que vai acontecer. Segurança por design.

🤖 Agent skills

Mais de 100 skills incluídas para LLMs. Cada skill descreve como usar determinado grupo de comandos — agentes leem skills e sabem operar Workspace sem tooling customizado.

O que é uma skill

Cada skill é um arquivo que descreve para um agente LLM como usar um grupo específico de comandos gws:

Descrição

O que a skill faz e quando usá-la

Comandos

Quais comandos gws usar e em que ordem

Exemplos

Inputs e outputs esperados para cada cenário

O conceito é poderoso: em vez de ensinar cada agente LLM como operar cada API do Google, as skills fornecem o "manual de instruções" em formato que LLMs entendem nativamente. O agente lê a skill, entende os comandos disponíveis e opera o Workspace sem código customizado.

💡 Analogia

Skills são como SKILL.md do CLI-Anything (Trilha 3): arquivos de contrato entre humano/agente e ferramenta. A diferença é que no gws, as skills já vêm incluídas — mais de 100 cobrindo todos os serviços do Workspace. O agente não precisa "aprender" a usar as APIs — ele lê a skill e sabe.

🔌 gws como extensão do Gemini CLI

Quando gws é instalado como extensão do Gemini CLI, o agente ganha acesso a todo o Google Workspace via linguagem natural. Exemplos reais:

"Agende uma reunião com o time para quinta às 14h"

→ Gemini CLI lê a skill de Calendar, monta o comando gws, cria o evento

"Encontre os relatórios de Q1 no Drive e compartilhe com marketing@"

→ Busca no Drive, identifica arquivos, configura permissões de compartilhamento

"Responda o email do João sobre o orçamento com os dados da planilha"

→ Lê o email no Gmail, acessa dados no Sheets, compõe e envia resposta

💡 O panorama completo

Gemini CLI + gws cria uma superfície onde todo o Google Workspace é operável via terminal com linguagem natural. Isso transforma o terminal de "ferramenta de desenvolvedor" em "centro de operações" para qualquer profissional que usa Google Workspace no dia a dia.
