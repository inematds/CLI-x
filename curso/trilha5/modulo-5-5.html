Projeto 5: Orquestrando multiplos agentes

🏗️ Arquitetura multi-agente

Um agente principal orquestra outros via MCP. Cada agente e especialista — o orquestrador decide quem faz o que.

Modelo mental

Agente Orquestrador

Claude Code / Gemini CLI

↓↓↓
MCP GitHub

Issues, PRs, code

MCP Notes

Notas, docs

MCP gws

Email, calendar

💡 Composicao de capacidades

Cada MCP server adiciona capacidades ao agente. Um server de GitHub permite operar repositorios. Um server de notas permite gerenciar conhecimento. Um server gws permite operar email e calendario. Juntos, o agente pode: ler uma issue, analisar codigo, propor fix, criar PR e notificar o time — tudo numa unica interacao.

🔗 Conectando MCP servers

Configure multiplos MCP servers no mesmo agente. O agente ve todas as tools de todos os servers simultaneamente.

Configuracao com 3 servers

// .claude/settings.json
{
  "mcpServers": {
    "github": {
      "command": "npx",
      "args": ["-y", "@modelcontextprotocol/server-github"],
      "env": {
        "GITHUB_TOKEN": "ghp_..."
      }
    },
    "notes": {
      "command": "npx",
      "args": ["tsx", "./mcp-notes/src/index.ts"]
    },
    "workspace": {
      "command": "npx",
      "args": ["-y", "@anthropic-ai/gws-mcp"]
    }
  }
}

📊 O que o agente ve

Com 3 servers conectados, o agente tem acesso a todas as tools simultaneamente:

GitHub (8 tools)

create_issue, create_pr, get_file, search_code...

Notes (4 tools)

create_note, list_notes, search_notes, delete_note

Workspace (12 tools)

gmail_send, drive_upload, sheets_read, calendar_create...

🔄 Workflow completo

Um pipeline de 7 passos que usa multiplos MCP servers em sequencia. De uma issue no GitHub ate a notificacao por email.

Pipeline: Issue → PR → Notificacao

1
Le issue no GitHub → MCP GitHub: get_issue
2
Analisa codigo relevante → MCP GitHub: search_code + get_file
3
Propoe fix → Agente (raciocinio interno)
4
Cria branch e commit → MCP GitHub: create_branch + push_files
5
Roda testes → Shell execution (npm test)
6
Cria Pull Request → MCP GitHub: create_pull_request
7
Notifica o time → MCP Workspace: gmail_send

Prompt que dispara o workflow

> Leia a issue #42 do repo my-org/my-app, analise o codigo,
  proponha um fix, crie uma PR, e envie email para o time
  em time@empresa.com com o link da PR

Um unico prompt. O agente orquestra 7 passos usando 3 MCP servers diferentes.

📊 Observabilidade

Sem observabilidade, debugging e impossivel. Voce precisa saber o que o agente fez, quanto custou e onde falhou.

Metricas essenciais

Tempo

Duracao por step

Tokens

Consumo por acao

Erros

Taxa de falha

Tools

Chamadas por server

Structured logging

// Log de cada acao do agente em JSON
{
  "timestamp": "2026-03-23T14:32:01Z",
  "step": 3,
  "server": "github",
  "tool": "create_pull_request",
  "duration_ms": 2340,
  "tokens_used": 1250,
  "status": "success",
  "result": { "pr_number": 87 }
}

📊 Ferramentas de observabilidade

Log files: salve cada sessao do agente em arquivo JSON rotacionado
Custo tracking: conte tokens por sessao para estimar custo (tokens x preco)
Dashboard: um script simples que le os logs e gera metricas por dia/semana

⚠️ Tratamento de erros

Agentes falham. APIs caem. Permissoes expiram. Lidar com falhas e tao importante quanto o fluxo principal.

Tipos de falha comuns

Timeout

MCP server demora mais que o limite

Estrategia: retry com backoff exponencial

API Error (4xx/5xx)

GitHub rate limit, Google quota excedida

Estrategia: esperar e retentar ou abortar

Permissao negada

Token expirado, scope insuficiente

Estrategia: fallback para humano

Output inesperado

Tool retorna formato diferente do esperado

Estrategia: validacao + skip e continuar

Padrao de retry com backoff

async function withRetry<T>(
  fn: () => Promise<T>,
  maxRetries = 3,
  baseDelay = 1000
): Promise<T> {
  for (let i = 0; i < maxRetries; i++) {
    try {
      return await fn();
    } catch (err) {
      if (i === maxRetries - 1) throw err;
      const delay = baseDelay * Math.pow(2, i);
      console.log(`Retry ${i+1}/${maxRetries} em ${delay}ms`);
      await new Promise(r => setTimeout(r, delay));
    }
  }
  throw new Error("Unreachable");
}

⚠️ Regra de ouro

Nunca ignore erros silenciosamente em producao. Cada falha deve ser logada, categorizada e — se possivel — alertada. Um agente que falha em silencio e pior que um agente que nao roda: ele da a falsa impressao de que tudo esta funcionando.

6

🚀 Producao

Agentes em producao precisam da mesma engenharia que microsservicos: idempotencia, rate limiting, circuit breakers e health checks.

Boas praticas para producao

Idempotencia

Executar o mesmo workflow duas vezes deve produzir o mesmo resultado. Antes de criar uma PR, verifique se ja existe uma para a mesma issue.

Rate limiting

Limite chamadas por minuto a cada API. GitHub: 5000/hora. Google: varia por servico. Respeite os limites ou sera bloqueado.

Circuit breakers

Se um MCP server falha 3 vezes seguidas, pare de chama-lo por 5 minutos. Evita cascata de falhas.

Health checks

Periodicamente verifique se cada MCP server esta respondendo. Alerte quando um server cair.

Health check simples

#!/bin/bash
# health-check.sh - rode via cron a cada 5 minutos

for server in github notes workspace; do
  if ! mcp-client ping "$server" --timeout 5000; then
    echo "[ALERT] MCP server '$server' nao respondeu" | \
      mail -s "MCP Health Check FAIL" ops@empresa.com
  fi
done

💡 Escalabilidade

Para escalar, rode multiplas instancias de agentes com filas de trabalho (Redis, SQS). Cada instancia puxa uma tarefa da fila, executa o workflow, e reporta o resultado. Isso e exatamente o padrao de workers/consumers de microsservicos — agentes nao sao diferentes.

✓ Producao exige engenharia real: idempotencia, rate limiting, circuit breakers e health checks
